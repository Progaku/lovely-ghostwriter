# Pull Request 作成

## 目的

現在のブランチのコミット履歴と差分を読み取り、レビュー担当者が実装内容、関連 issue、確認済み事項、残るリスクを判断できる pull request を日本語で作成する。

PR本文は、実装内容の列挙だけでなく、「なぜ必要か」「何を確認したか」「何を対象外にしたか」「安全制約を壊していないか」を含める。

## 基本手順

1. `AGENTS.md` を読み、言語方針、技術スタック、安全制約を確認する。
2. `git status --short` と `git branch --show-current` で、現在の作業ツリーとブランチを確認する。
3. ベースブランチを特定する。分からない場合は `main`、`master`、`develop`、リモート既定ブランチの候補を確認し、推測でPRを作成しない。
4. ベースブランチとの差分、コミット履歴、変更ファイル一覧を確認する。
5. ブランチ名、コミットメッセージ、差分、PR本文の既存案から関連 issue を特定する。
6. GitHub issue が確認できる場合は `gh issue view` で本文、受け入れ条件、対象外、URLを読む。
7. `docs/` 配下のうち、今回の変更に関係する要件や設計を確認する。
8. 実行済みの検証が分からない場合は、`package.json` の scripts を確認し、必要な検証コマンドを実行する。
9. PRタイトルとPR本文を日本語で作成する。
10. ユーザーが「GitHubに作成して」「PRを作って」など実作成を明示した場合のみ、`gh pr create` を実行する。明示がない場合はPR本文のドラフトを提示する。

## Git 確認

まず作業ツリーとブランチを確認する。

```bash
git status --short
git branch --show-current
git remote -v
```

ベースブランチとの差分確認の例:

```bash
git merge-base HEAD origin/main
git log --oneline origin/main..HEAD
git diff --stat origin/main...HEAD
git diff origin/main...HEAD
```

リモート既定ブランチが不明な場合は、必要に応じて確認する。

```bash
git remote show origin
```

ネットワークが必要な `git fetch`、`git remote show origin`、`gh` コマンドが失敗した場合は、許可を取って再実行する。失敗したまま推測でGitHub上のPRを作成しない。

## 関連 issue の扱い

PRは原則として GitHub issue に紐づける。

確認する内容:

- issue 番号、タイトル、URL、状態。
- 背景、やること、受け入れ条件、テスト観点、対象外。
- 関連する `docs/`、`references/`、実装ファイル。
- 非公式ファン作品、娯楽用途、重要判断に使わない注意書きなどの安全制約。
- 初期スコープ外の保存、履歴、URL共有、画像保存、SNS連携、認証、バックエンド、外部 AI API 呼び出しが混入していないか。

関連 issue が確認できない場合は、PR本文に「関連 issue: 未確認」と明記し、レビュー時のリスクとして残す。issue が必要な運用であれば、PR作成前にユーザーへ確認する。

## 検証

`package.json` を確認し、存在する script だけを実行する。

```bash
npm run lint
npm run test
npm run build
```

すべてを機械的に実行するのではなく、変更範囲に応じて必要な検証を選ぶ。UI変更がある場合は、主要画面幅、テキスト折り返し、focus、disabled、error状態などの手動確認も本文に含める。

検証できなかった場合は、未実行の理由と残るリスクをPR本文に書く。

## PR本文

PR本文は日本語で、レビューできる粒度にする。

標準フォーマット:

```markdown
## 概要

- <変更の目的と背景>
- <主要な実装内容>

## 関連 issue

- Closes #<number>

## 実装内容

- <変更1>
- <変更2>
- <変更3>

## 確認したこと

- `<command>`: 成功
- <手動確認>: 成功

## 対象外

- <今回あえて含めていないこと>

## レビュー観点

- <重点的に見てほしい仕様やリスク>
```

issue を閉じる意図が明確な場合のみ `Closes #<number>` を使う。紐づけだけの場合は `Refs #<number>` にする。

## PRタイトル

タイトルは日本語で、何を実装・修正したPRかが分かる短い文にする。

例:

- `週ごとの予言生成フォームを追加`
- `コピー用AIプロンプトの生成ロジックを実装`
- `非公式ファン作品の注意書きを表示`

issue番号を付ける既存運用がある場合は合わせる。分からない場合は本文の関連 issue に番号を書く。

## GitHub PR を作成する場合

ユーザーが実作成を明示した場合だけ、GitHub CLI の状態を確認してPRを作成する。

```bash
gh auth status
gh pr create --base <base-branch> --head <current-branch> --title "<タイトル>" --body-file "<本文ファイル>"
```

本文ファイルは一時ファイルとして作成してよい。作成後はPR番号とURLをユーザーへ伝える。

`gh pr create` の前に確認すること:

- 現在ブランチがPR対象ブランチである。
- 必要なコミットが含まれている。
- ベースブランチが正しい。
- 未コミット変更がPRに含まれないまま残っている場合、その扱いをユーザーへ確認済みである。
- PR本文に関連 issue、実装内容、検証、対象外、レビュー観点が含まれている。

## 注意事項

- 未コミット変更を破棄、退避、上書き、巻き戻ししない。
- `git reset --hard`、`git checkout --`、`git clean` などの破壊的操作は行わない。
- PR本文で、安全制約や非公式ファン作品の注意書きを弱める表現をしない。
- 公式画像、漫画コマ、アニメ素材、既存キャラクター画像の使用を前提にした説明を書かない。
- 予言文や生成例を説明する場合、現実の危害、病気、死、犯罪、金銭損失などを断定的に予告する表現を避ける。

## 報告形式

PRドラフトを提示した場合:

```markdown
現在のブランチ `<branch>` と `<base>` との差分から、PR本文ドラフトを作成しました。

タイトル:
<タイトル>

本文:
<PR本文>

検証:
- `<command>`: 成功
- `<command>`: 未実行（理由）
```

GitHub上に作成した場合:

```markdown
PRを作成しました。

- <PR URL>
- タイトル: <タイトル>
- ベース: `<base>`
- ヘッド: `<branch>`

検証:
- `<command>`: 成功
```
