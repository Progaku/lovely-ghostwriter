# Issue 実装

## 目的

指定された issue を唯一の作業範囲として扱い、設計ドキュメントとコーディング規約に沿って実装、テスト、検証まで完了する。

実装中は、ユーザーの未コミット変更を尊重し、issue の対象外や安全制約を越える変更を混ぜない。

## 基本手順

1. `AGENTS.md` を読み、言語方針、技術スタック、安全制約を確認する。
2. ユーザーが指定した issue 番号、URL、または本文から対象 issue を特定する。
3. GitHub issue の場合は `gh issue view` でタイトル、本文、URL、状態、ラベルを確認する。
4. issue の背景、やること、受け入れ条件、テスト観点、対象外、参照を要約する。
5. `docs/` 配下の要件、詳細設計、テンプレート方針など、issue に関係する箇所を読む。
6. 実装前に `references/coding.md` を読み、該当する実装観点を確認する。
7. `git status --short` と現在ブランチを確認し、未コミット変更を上書きしない。
8. 関連する既存実装、テスト、設定ファイルを調査して、既存パターンに合わせた実装方針を決める。
9. issue の受け入れ条件を満たす最小範囲でコード、テスト、必要なドキュメントを更新する。
10. 利用可能な検証コマンドを確認して実行し、結果と未確認リスクを記録する。
11. 変更内容、満たした受け入れ条件、実行した検証、残課題を日本語で報告する。

## issue 確認

GitHub CLI が使える場合は、次を基本にする。

```bash
gh issue view <number-or-url> --json number,title,body,state,url,labels,assignees,milestone
```

GitHub CLI が使えない、ネットワークが必要な操作が失敗する、または issue 本文が不足している場合は、推測で実装しない。ユーザー提供の issue 本文か、追加確認を根拠にする。

確認する内容:

- issue 番号、タイトル、URL、状態。
- 背景、やること、受け入れ条件、テスト観点、対象外。
- `docs/`、`references/`、関連ファイルへの参照。
- 非公式ファン作品、娯楽用途、重要判断に使わない注意書きなどの安全制約。
- 初期スコープ外の保存、履歴、URL共有、画像保存、SNS連携、認証、バックエンド、外部 AI API 呼び出しが含まれていないか。

issue が closed、内容不足、または `AGENTS.md` や `docs/` と矛盾する場合は、実装前にユーザーへ確認する。

## 実装方針

- issue の対象外を実装しない。関連して見つけた改善点は、必要なら報告に留める。
- UI文言、生成される予言文、バリデーションメッセージ、AI用プロンプト、ユーザー向けドキュメントは日本語にする。
- コード上の識別子、型名、関数名、ファイル名、設定キー、テスト名は TypeScript/React/Vite の慣習に従って英語にしてよい。
- 共通化は、実際の重複や複雑さを減らす場合に限る。
- 新しい依存を追加する前に、既存依存、標準API、簡潔な自前実装で足りるか確認する。
- React コンポーネントは表示、状態管理、ドメインロジックを混ぜすぎない。複雑な変換や生成ロジックは純粋関数へ分離する。
- フォーム入力、validation、生成ロジックは、UI表示とテストから参照しやすい形にする。
- アクセシビリティを実装範囲に含める。入力label、エラーメッセージ関連付け、keyboard操作、focus表示、disabled状態を確認する。
- 外部API送信、永続化、個人情報のログ出力など、初期スコープ外または安全制約に反する処理を追加しない。

詳細な実装基準は `references/coding.md` を正本として扱う。判断に迷う場合は、一般論ではなく該当見出しを読み返して決める。

## テストと検証

まず `package.json` を確認し、存在する script だけを実行する。

```bash
npm run lint
npm run test
npm run build
```

実装内容に応じて、次を追加または更新する。

- 純粋関数には Vitest の単体テスト。
- React コンポーネントには Testing Library でユーザー操作と表示結果のテスト。
- validation、プロンプト生成、予言文生成、安全制約に関わる分岐には境界値や回帰テスト。
- UI変更には主要な画面幅、テキスト折り返し、focus、disabled、error状態の手動確認。

検証コマンドが存在しない、依存不足、環境制約で実行できない場合は、未実行の理由と残るリスクを報告する。

## Git と作業ツリー

作業前後に次を確認する。

```bash
git status --short
git branch --show-current
```

- 未コミット変更を破棄、退避、上書き、巻き戻ししない。
- `git reset --hard`、`git checkout --`、`git clean` などの破壊的操作を行わない。
- ユーザーの変更と同じファイルを編集する必要がある場合は、内容を読んで共存できる形で最小変更にする。
- ブランチ作成が必要な場合は、`issue-branch-create` の方針に合わせる。ユーザーが実装のみを求めている場合、勝手にブランチを作成しなくてもよい。

## ドキュメント更新

実装判断が現在の設計から意図的に外れる場合は、関連する `docs/` を更新する。

ただし、非公式ファン作品の注意書き、安全制約、娯楽用途、重要判断に使わない方針を削除または弱体化しない。生成例やテンプレートは、現実の危害や断定的予言を避ける方針と一致させる。

## 完了報告

報告は日本語で簡潔に書く。

```markdown
issue #<number>「<タイトル>」の実装を完了しました。

変更内容:
- <主要変更1>
- <主要変更2>

検証:
- `<command>`: 成功
- `<command>`: 未実行（理由）

補足:
- <残課題や注意点があれば>
```

issue を完了できなかった場合は、どこまで確認・実装したか、ブロッカー、次に必要な判断を明記する。
