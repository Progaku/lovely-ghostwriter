# ラブリーゴーストライター Issue 作成

## 目的

ラブリーゴーストライター風 Web アプリの開発を issue 駆動で進めるため、設計ドキュメントに根拠を持つ実装可能な issue を日本語で作成する。

issue は「何を作るか」だけでなく、「なぜ必要か」「完了条件」「確認方法」「安全制約」を含める。

## 基本手順

1. `AGENTS.md` を読み、プロジェクトの言語方針と安全制約を確認する。
2. `docs/*` の中にある、issue の対象に関係する箇所を読む。
3. 既存実装がある場合は、関連する `src/`、`test`、設定ファイルを確認して、未実装・差分・リスクを特定する。
4. issue を実装単位へ分ける。1つの issue は、1回の実装 PR で完了できる粒度にする。
5. 各 issue に受け入れ条件、テスト観点、対象外、注意事項を入れる。
6. ユーザーが「GitHub に作成して」と明示した場合のみ `gh issue create` などで実作成する。明示がない場合は issue 本文のドラフトを出す。

## 分割方針

- 仕様、UI、生成ロジック、機能、テスト、ドキュメント更新は原則として別 issue に分ける。
- 共通型や定数など、後続 issue の土台になる作業は先行 issue として切る。
- 1つの issue に複数の独立機能を混ぜない。
- 完了条件が曖昧な調査は、成果物を「判断メモ」「設計更新」「実装方針の確定」などに固定する。
- 初期スコープ外の保存、履歴、URL共有、画像保存、SNS連携、認証、バックエンド、外部 AI API 呼び出しは issue に含めない。必要なら「対象外」として明記する。

## 必須制約

- issue のタイトル、本文、受け入れ条件、テスト観点は日本語にする。
- UI文言、予言文、バリデーションメッセージ、AI用プロンプトは日本語のみで扱う。
- 非公式ファン作品、娯楽用途、重要判断に使わない注意書きを削除・弱体化しない。
- 公式画像、漫画コマ、アニメ素材、既存キャラクター画像の使用を前提にしない。
- 予言文や生成例では、現実の危害、病気、死、犯罪、金銭損失などを断定的に予告しない。
- 専門判断が必要な医療、法律、金融などは専門家への相談を促す方針に沿わせる。
- React、Vite、TypeScript、Tailwind CSS、MUI、Vitest の構成から意図的に外れる場合は、確認を挟んだ上で許可が出た場合のみ、関連ドキュメント更新を issue に含める。

## issue 本文

標準テンプレートは `references/issue-template.md` を使う。複数 issue をまとめて作るときも、各 issue に同じ見出しを使う。

最低限、以下を含める。

- 背景
- やること
- 受け入れ条件
- テスト観点
- 対象外
- 参照

## GitHub issue を作成する場合

ユーザーが実作成を求めた場合だけ、GitHub CLI の状態を確認する。

```bash
gh auth status
gh issue create --title "<タイトル>" --body-file "<本文ファイル>"
```

本文ファイルは一時ファイルとして作成してよい。作成後は issue 番号と URL をユーザーへ伝える。

ラベル、マイルストーン、assignee は、既存運用が確認できる場合のみ合わせる。分からない場合は付けないか、ドラフト内に候補として書く。

## 品質チェック

作成前または提示前に、次を確認する。

- issue 単体で実装者が着手できる。
- 受け入れ条件がチェック可能な文になっている。
- テスト観点が Vitest、手動確認、レスポンシブ確認などに分かれている。
- 要件・詳細設計・テンプレート方針への参照がある。
- 安全制約と初期スコープ外を破っていない。
- issue 間に依存関係がある場合、順序が分かる。
