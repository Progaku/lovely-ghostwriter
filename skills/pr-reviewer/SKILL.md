# Pull Request レビュー

## 目的

pull request が関連 issue に基づいており、issue の受け入れ条件を満たし、プロジェクトの決まり事と安全制約を守った実装になっているかを確認する。

レビューは、好みの指摘ではなく、ユーザー影響、仕様逸脱、保守性、セキュリティ、テスト不足に集中する。問題点だけでなく、PRのよい点も具体的に挙げる。

## 基本手順

1. `AGENTS.md` を読み、言語方針、技術スタック、安全制約を確認する。
2. PR本文、ブランチ名、コミット、差分から関連 issue を特定する。
3. 関連 issue が確認できない場合は、最低条件未達として最重要指摘にする。推測で issue があるものとして扱わない。
4. issue 本文を読み、背景、やること、受け入れ条件、テスト観点、対象外を把握する。
5. `docs/` 配下の要件、詳細設計、テンプレート、注意書きに関係する箇所を読む。
6. コーディングに関する実装観点を確認するため、リポジトリ直下の `references/coding.md` を読む。
7. PR差分を読み、実装が issue、docs、`references/coding.md` の要求を満たしているか確認する。
8. テスト、型チェック、lint、ビルドの実行結果を確認する。未実行の場合は、必要性と未確認リスクを明記する。
9. 指摘は重大度順に、日本語で具体的なファイル・行番号・再現条件・修正方針を添えて出す。
10. 指摘だけでなく、設計・実装・テスト・文言・安全制約への配慮など、PRのよい点を具体的に挙げる。

## GitHub PR を確認する場合

ユーザーが PR 番号、URL、または現在のブランチのレビューを求めた場合は、可能な範囲で GitHub CLI と git を使う。

```bash
gh pr view <number-or-url> --json number,title,body,url,headRefName,baseRefName,closingIssuesReferences,commits,files
gh pr diff <number-or-url>
git status --short
git diff --stat <base>...<head>
git diff <base>...<head>
```

GitHub CLI が使えない場合は、ローカル差分、PR本文の貼り付け、ユーザー提供情報から確認する。関連 issue が確認できないままなら、その事実をレビュー結果に残す。

## 最低条件

- PRには GitHub issue が紐づいていること。
- issue なし、または issue の内容が確認できない場合は、`blocking` の指摘にする。
- PR本文やコミットに issue 番号が書かれていても、issue本文を確認できない場合は「issue充足は未確認」と扱う。
- issue と無関係な機能追加、仕様変更、リファクタリングが混ざっている場合は、分割または issue 更新を求める。

## レビュー観点

### issue と仕様

- issue の受け入れ条件をすべて満たしているか。
- issue の対象外を実装していないか。
- `docs/*`、関連テンプレートと矛盾しないか。
- 意図的に設計から外れる変更がある場合、関連ドキュメントも更新されているか。
- 非公式ファン作品、娯楽用途、重要判断に使わない注意書きが削除・弱体化されていないか。

### コーディング規約

- 実装品質、TypeScript、React、状態管理、Vite、スタイリング、アクセシビリティ、フォーム、外部I/O、命名、import/export、エラーハンドリング、テスト、パフォーマンス、セキュリティの詳細観点は `references/coding.md` を正本として確認する。
- `references/coding.md` と差分実装が矛盾する場合は、該当見出し名と根拠を添えて指摘する。
- 実装観点の細目を記憶や一般論だけで補完しない。判断に迷う場合は `references/coding.md` の該当箇所を読み返す。

### 日本語とプロダクト方針

- UI文言、生成される予言文、バリデーションメッセージ、AI用プロンプト、ユーザー向け文書が日本語になっているか。
- 予言文や生成例が、現実の危害、病気、死、犯罪、金銭損失などを断定的に予告していないか。
- 医療、法律、金融など専門判断が必要なテーマで、専門家への相談を促す方針に反していないか。
- 公式画像、漫画コマ、アニメ素材、既存キャラクター画像の使用を前提にしていないか。

### セキュリティと安全性

- 外部API呼び出し、ネットワーク送信、認証、保存機能など、初期スコープ外の処理が混入していないか。
- クリップボード、localStorage、URL、ログに不要な個人情報や相談内容を残していないか。
- 生成例やテンプレートが、安全制約に反する断定的・危害的な内容を含んでいないか。
- 一般的な実装セキュリティの観点は `references/coding.md` の「セキュリティ」を確認する。

## 推奨コマンド

必要に応じて、既存の `package.json` を確認してから実行する。

```bash
npm test
npm run lint
npm run typecheck
npm run build
```

存在しないコマンドを推測で実行しない。該当スクリプトがない場合は、利用可能な検証コマンドと未確認リスクをレビュー結果に書く。

## 出力形式

レビュー結果は日本語で書く。コードレビューとして、指摘を先に出し、要約は最後に短く置く。

重大度は次を使う。

- `blocking`: マージ前に必須で直す。issue未紐づけ、issue未充足、重大な不具合、セキュリティ、安全制約違反。
- `major`: 直すべき。仕様漏れ、テスト不足、保守性やUXに明確な問題がある。
- `minor`: 直すとよい。小さな一貫性、可読性、文言、軽微な改善。

レビューコメントの種別は、必要に応じて重大度の後ろに併記する。

- `must`: マージ前に対応が必要。
- `should`: 対応を強く推奨する。
- `nits`: 軽微な修正提案。
- `imo`: レビュー担当者の意見としての提案。
- `fyi`: 参考情報、将来の注意点。

問題点とは別に、PRのよい点を `Good Points` に書く。よい点は抽象的に褒めず、どの実装・設計・テスト・文言・安全配慮が有効だったかを具体的に書く。

標準フォーマット:

```markdown
**Findings**
- `blocking` `must` [path/to/file.tsx:42] 指摘内容。なぜ問題か、どのissue条件に反するか、どう直すべきか。
- `major` `should` [path/to/test.ts:18] 指摘内容。

**Good Points**
- [path/to/file.tsx:24] よい点。どの観点で有効かを具体的に書く。

**Open Questions**
- 確認が必要な点があれば書く。なければ省略する。

**Verification**
- 実行したコマンドと結果を書く。
- 実行できなかった検証と理由を書く。

**Summary**
- PRがissueを満たしているか、マージ可能かを短く結論づける。
```

問題が見つからない場合も、関連 issue、確認した差分、実行した検証、残るリスクを明記する。
