# ラブリーゴーストライター風 Webアプリ 実装仕様

## 1. 目的

`./requirements.md` をもとに、React/Vite/TypeScript で初期実装するための仕様を定義する。

本仕様は、フロントエンド単体SPAとして動作する初期版を対象にする。バックエンド、DB、認証、外部AI API連携、永続保存、共有機能は含めない。

## 2. 実装スコープ

### 2.1 実装するもの

- 日本語UIの単一ページアプリ
- 入力フォーム
- 入力バリデーション
- 4週分、各週1行の予言テキスト生成
- 入力値由来のシードによる再生成
- AI用プロンプト生成
- AI用プロンプトのコピー
- コピー結果のフィードバック表示
- 非公式ファン作品、娯楽用途、重要判断非推奨の注意書き
- スマートフォンとデスクトップに対応したレスポンシブ表示

### 2.2 実装しないもの

- 外部AI API呼び出し
- サーバーサイド処理
- データベース保存
- 履歴表示
- URL共有
- 画像保存
- SNS投稿
- ユーザーアカウント
- 多言語対応
- 公式画像、漫画コマ、アニメ素材、既存キャラクター画像の使用

### 3.1 技術スタック

- React
- Vite
- TypeScript
- MUI

- アラート

## 4. データ型

### 4.1 入力データ

```ts
export type Gender = "男性" | "女性" | "その他" | "";

export type ProphecyInput = {
  name: string;
  birthDate: string;
  gender: Gender;
  theme: string;
  mood: string;
};
```

### 4.2 生成結果

```ts
export type ProphecyWeek = {
  weekNumber: 1 | 2 | 3 | 4;
  line: string;
  candidateId: string;
  lineKey: "line1" | "line2" | "line3" | "line4";
  selectedTerms: {
    symbol?: string;
    place?: string;
    object?: string;
    action?: string;
    toneHint?: string;
    adviceNuance?: string;
  };
};

export type ProphecyResult = {
  weeks: ProphecyWeek[];
  interpretationAxis: string;
  aiPrompt: string;
  seed: string;
  randomSalt: string;
  generatedAt: string;
};
```

`seed` は、正規化した入力値から生成したシード文字列を保持する。画面表示は必須ではないが、入力内容が生成結果へ反映されたことをテストしやすくするために保持する。

`randomSalt` は、生成操作ごとに作るランダムな文字列を保持する。同じ入力でも再生成で表現を変えるため、最終的な疑似乱数の初期化には `seed` と `randomSalt` の両方を使う。

`generatedAt` は画面表示必須ではないが、結果生成のタイミングを確認できるよう保持する。予言本文とAI用プロンプトの内容は `generatedAt` に依存させない。

`candidateId`、`lineKey`、`selectedTerms` は画面表示必須ではない。AI用プロンプトに含める `interpretationAxis` を、実際に選ばれた4行と語彙から組み立てるために保持する。

`interpretationAxis` は、固定の分類名ではなく、4週分の `line`、各週の候補 ID、行番号ごとの基本メタ情報、差し込まれた語彙、週ごとの助言ニュアンスを要約した文字列にする。

### 4.3 バリデーションエラー

```ts
export type ValidationErrors = Partial<Record<keyof ProphecyInput, string>>;
```

## 5. 画面仕様

### 5.1 画面全体

単一ページ構成にする。

表示順は以下とする。

1. タイトル領域
2. 注意書き
3. 入力フォーム
4. 生成結果
5. AI用プロンプト

結果が未生成の場合、4と5は表示しない。

### 5.2 タイトル領域

以下を表示する。

- アプリ名
- 短い補足文

文言例:

- アプリ名: `Lovely Ghostwriter`
- 補足文: `各週1行で4週分の予言文となる四行詩をつくる非公式ファンWebアプリ`

公式作品と誤認される表現は避ける。

実装はMUIの `Typography` とTailwind CSSの余白・配置ユーティリティを組み合わせる。

### 5.3 注意書き

画面上部またはフォーム付近に、最小限の注意書きを表示する。

必須文言:

```text
このアプリは非公式ファン作品です。生成結果は娯楽用途の文章であり、重要な判断には使用しないでください。
```

実装はMUIの `Alert` を使う。

### 5.4 入力フォーム

#### 5.4.1 項目

| 項目 | UI | 必須 | 上限 | 備考 |
| --- | --- | --- | --- | --- |
| 名前 | MUI `TextField` | 必須 | 50文字 | 前後空白はバリデーション時に除去 |
| 生年月日 | MUI `TextField` | 必須 | - | `type="date"` を使う |
| 性別 | MUI `Select` | 任意 | - | 未選択、男性、女性、その他 |
| 相談テーマ | MUI `TextField` | 必須 | 255文字 | `multiline` を使う |
| 今月の気分 | MUI `TextField` + 選択肢ボタン | 必須 | 255文字 | プリセット選択と自由入力の両方に対応する |

#### 5.4.2 入力補助

- 各入力には日本語のラベルを付ける。
- 必須項目は視覚的に分かるようにする。
- 上限のある項目は `maxLength` を設定する。
- テキストエリアは長文でもレイアウトが崩れない幅にする。

今月の気分には、入力欄の近くにプリセット選択肢を表示する。選択肢を押すと、今月の気分の入力値をその選択肢の文言で更新する。更新後も `TextField` は編集可能なままにし、ユーザーが自由入力で上書きできるようにする。

プリセット選択肢の初期案:

- 静かに過ごしたい
- 少し不安がある
- 変化を求めている
- 背中を押されたい
- 人との距離を考えている
- 集中したい
- 迷いを整理したい

実装はMUIの `Button`、`Chip`、または `ToggleButton` を使う。単一選択として扱い、選択済みの文言と入力欄の値が一致する場合は選択状態を視覚的に示してよい。ただし、自由入力値がプリセットに一致しない場合でもエラーにはしない。

#### 5.4.3 操作ボタン

初回生成前:

- `予言を生成する`

生成後:

- `再生成する`

必須入力が未入力、またはバリデーションエラーがある場合、生成ボタンと再生成ボタンは押せない状態にする。

ボタンの `disabled` は、現在の入力値をバリデーションした結果で制御する。押下後に初めて無効化するのではなく、入力中の状態に追従して有効/無効を切り替える。

実装はMUIの `Button` を使う。横幅や余白はTailwind CSSで調整する。

### 5.5 結果表示

生成後に、4週分を週ごとに表示する。

各週の表示要素:

- `第1週` から `第4週` の見出し
- 1行の詩的な予言文

読みやすい余白で表示する。各週の予言文は個別の `<p>` として出力してよい。

週ごとの表示はMUIの `Paper` または `Box` を使う。背景色、影、余白、レスポンシブの並びはTailwind CSSで調整してよい。

### 5.6 AI用プロンプト表示

生成後に、AI用プロンプトを表示する。

表示要素:

- 見出し
- コピーボタン
- プロンプト本文
- コピー成功または失敗の短いフィードバック

プロンプト本文は手動コピーできるように、`textarea` または選択可能なテキスト領域で表示する。

コピー結果の表示はMUIの `Snackbar`、`Alert`、またはインラインの `Typography` を使う。

## 6. 状態管理仕様

`App.tsx` で以下の状態を管理する。

```ts
const [input, setInput] = useState<ProphecyInput>(initialInput);
const [errors, setErrors] = useState<ValidationErrors>({});
const [result, setResult] = useState<ProphecyResult | null>(null);
const [copyStatus, setCopyStatus] = useState<"idle" | "success" | "error">("idle");
```

生成ボタンの有効/無効は、現在の入力値から派生する。

```ts
const currentErrors = validateProphecyInput(input);
const isFormValid = Object.keys(currentErrors).length === 0;
```

`isFormValid` が `false` の場合、生成ボタンと再生成ボタンは `disabled` にする。

### 6.1 状態遷移

#### 入力変更

- 対象項目の値を更新する。
- 入力値から最新のバリデーション結果を算出し、生成ボタンの有効/無効に反映する。
- コピー状態は変更しない。
- 既存結果は自動では消さない。

#### 今月の気分プリセット選択

- 選択されたプリセット文言を `input.mood` に設定する。
- 設定後、最新のバリデーション結果を算出し、生成ボタンの有効/無効に反映する。
- `input.mood` は通常のテキスト入力と同じ値として扱う。
- 選択後もユーザーが `input.mood` を自由に編集できる。

#### 生成ボタン押下

生成ボタンは有効な入力状態でのみ押下できる。

1. 念のため入力値を再バリデーションする。
2. エラーがあれば `errors` を更新し、`result` は変更しない。
3. エラーがなければ予言を生成する。
4. 生成した4週分の詩とメタ情報から、四行詩の解釈軸を生成する。
5. 生成した予言と四行詩の解釈軸からAI用プロンプトを生成する。
6. `result` を更新する。
7. `errors` を空にする。
8. `copyStatus` を `idle` に戻す。

#### 再生成ボタン押下

再生成ボタンも、現在の入力が有効な場合のみ押下できる。処理内容は生成ボタン押下と同じ。

生成ロジックでは現在の入力値からシードを作る。さらに、生成操作ごとにランダムな `randomSalt` を作り、`seed` と `randomSalt` を組み合わせて疑似乱数を初期化する。

同じ入力で再生成した場合、入力由来の `seed` は同じになるが、`randomSalt` が変わるため結果も変わり得る。入力値を変更した場合は `seed` も変わり、結果も変わり得る。

#### コピーボタン押下

1. `navigator.clipboard.writeText(result.aiPrompt)` を呼ぶ。
2. 成功時は `copyStatus` を `success` にする。
3. 失敗時は `copyStatus` を `error` にする。
4. 失敗時もプロンプト本文は選択可能なままにする。

## 7. バリデーション仕様

### 7.1 共通ルール

- 必須項目は `trim()` 後に空文字ならエラー。
- 上限文字数を超える入力は、HTML属性とバリデーション関数の両方で防ぐ。
- エラーメッセージは項目付近に表示する。
- エラーがある状態では生成ボタンと再生成ボタンを `disabled` にする。
- 生成処理側でも念のためバリデーションし、エラーがある場合は生成処理を実行しない。

## 8. 予言生成仕様

### 8.1 生成方式

外部AI APIは使わず、アプリ内のテンプレートと語彙リストを組み合わせて生成する。

生成関数:

```ts
export function generateProphecy(input: ProphecyInput): ProphecyWeek[];
```

四行詩の解釈軸生成関数:

```ts
export function buildInterpretationAxis(input: ProphecyInput, weeks: ProphecyWeek[]): string;
```

シード生成関数:

```ts
export function createProphecySeed(input: ProphecyInput): string;
```

シード付き疑似乱数生成関数:

```ts
export function createSeededRandom(seed: string, randomSalt: string): () => number;
```

### 8.2 出力制約

- 常に4週分を返す。
- `weekNumber` は1から4まで順に設定する。
- 各週の `line` は必ず1つの文字列にする。
- 第1週の本文は、`line1` 候補から選ぶ。
- 第2週から第4週の本文は、直前に選ばれた候補の `nextCandidates` から貪欲に選ぶ。
- 第1週から第4週までの組み合わせは固定しないが、各行を完全に独立したランダム選択にはしない。
- 各週には、選ばれた候補ID、行番号、差し込んだ語彙、助言ニュアンスをメタ情報として保持する。
- 各週の `line` は日本語文字列にする。
- 空行は生成しない。
- 危害、病気、死、犯罪、金銭損失を断定的に予告する語句は使わない。

### 8.3 文章トーン

象徴的、不穏、暗示的な雰囲気にする。ただし、現実の被害や重大な不幸を断定しない。

推奨する語彙:

- 紙
- インク
- 窓
- 鍵
- 月
- 影
- 針
- 机
- 封筒
- 静けさ
- 足音
- 余白

避ける語彙:

- 死ぬ
- 殺される
- 病気になる
- 破産する
- 犯罪に巻き込まれる
- 事故に遭う
- 必ず失敗する

### 8.4 テンプレート例

以下のような週番号別の行候補から、入力値由来のシードと生成操作ごとのランダム値で初期化した疑似乱数を使って選択する。

固定の4週セットを1つ選ぶのではなく、`line1` の行候補から第1週の候補を選び、その候補が持つ `nextCandidates` をたどって第2週以降を貪欲に選ぶ。第1週は `T01-L1-A`、第2週は `T10-L2-A` のように同じ月の中で候補を混ぜられるが、第2週以降は直前の行候補が指定する自然な候補群から選ぶ。

```ts
type TemplateLineKey = "line1" | "line2" | "line3" | "line4";

type TemplateLineCandidateRef = {
  lineKey: TemplateLineKey;
  candidateId: string;
};

type TemplateLineCandidate = {
  candidateId: string;
  lineKey: TemplateLineKey;
  text: string;
  profileId: string;
  nextCandidates?: TemplateLineCandidateRef[];
};

type TemplateLineCatalog = {
  // line1からline4は、第1週から第4週に対応する候補。
  line1: TemplateLineCandidate[];
  line2: TemplateLineCandidate[];
  line3: TemplateLineCandidate[];
  line4: TemplateLineCandidate[];
};

const templateLineCatalog: TemplateLineCatalog = {
  line1: [
    {
      candidateId: "T01-L1-A",
      lineKey: "line1",
      text: "{name}の名を薄いインクが覚え、{theme}の余白に小さな{symbol}が残る",
      profileId: "VP01",
      nextCandidates: [
        { lineKey: "line2", candidateId: "T01-L2-A" },
        { lineKey: "line2", candidateId: "T10-L2-A" },
      ],
    },
  ],
  line2: [
    {
      candidateId: "T01-L2-A",
      lineKey: "line2",
      text: "{theme}へ向かう行間に小さな{object}が挟まれ、{name}は順番を{action}",
      profileId: "VP01",
      nextCandidates: [
        { lineKey: "line3", candidateId: "T01-L3-A" },
      ],
    },
  ],
  line3: [
    {
      candidateId: "T01-L3-A",
      lineKey: "line3",
      text: "{mood}の気配は古い活字の間で滲み、{theme}の読み方を少し変える",
      profileId: "VP17",
      nextCandidates: [
        { lineKey: "line4", candidateId: "T01-L4-A" },
      ],
    },
  ],
  line4: [
    {
      candidateId: "T01-L4-A",
      lineKey: "line4",
      text: "最後の罫線を越える前に、{name}は手元の言葉を{action}とよい",
      profileId: "VP01",
    },
  ],
};
```

各週の候補文は1行だけで成立する文にする。名前、相談テーマ、今月の気分、小さな行動への余地は、特定の行だけに固定せず、各週の候補内へ必要に応じて自然に含める。週番号は表示上の見出しで示し、詩本文には含めない。

週ごとに選んだ `candidateId` は `ProphecyWeek` に保持する。`interpretationAxis` は、4週それぞれで実際に選ばれた候補 ID、行番号、行番号ごとの基本メタ情報、語彙、助言ニュアンスをもとに組み立てる。

### 8.5 シード付きランダム生成

入力内容に応じて表現を変えるため、以下のいずれかをシード付き疑似乱数で選択する。

- 週番号別の行候補選択
- 象徴語の選択
- 場所語の選択
- 道具語の選択
- 穏当な行動語の選択
- 行候補ごとのニュアンス
- 行末表現
- 週ごとの助言ニュアンス

選択した語彙とニュアンスは、詩本文へ差し込んだ後も `selectedTerms` に保持する。`interpretationAxis` は、この `selectedTerms` と4週分の `line` をもとに組み立てる。

入力値由来のシードは、生成に使う入力値を正規化してから作る。

正規化対象:

- `name.trim()`
- `birthDate`
- `gender`
- `theme.trim()`
- `mood.trim()`

シード文字列は、正規化した値を安定した順序で連結し、文字列ハッシュ関数に渡して作る。区切り文字は入力値と衝突しにくい固定文字列を使う。

例:

```ts
const seedSource = [
  input.name.trim(),
  input.birthDate,
  input.gender,
  input.theme.trim(),
  input.mood.trim(),
].join("\u001f");
```

再生成時のランダム性を担保するため、生成操作ごとに `randomSalt` を作る。`randomSalt` は `crypto.getRandomValues()` が使える場合はそれを優先し、使えない場合は `Math.random()` で作ってよい。

最終的な疑似乱数生成器の初期値は、入力値由来の `seed` と生成操作ごとの `randomSalt` を連結した文字列から作る。

例:

```ts
const rng = createSeededRandom(seed, randomSalt);
```

暗号学的な乱数品質は不要。軽量な文字列ハッシュと疑似乱数生成器を自前実装してよい。

実装候補:

- 文字列ハッシュ: FNV-1a、cyrb53 など
- 疑似乱数生成器: mulberry32、xorshift32 など

同じ正規化済み入力からは同じ `seed` を作ること。ただし、通常の生成処理では生成操作ごとに異なる `randomSalt` を混ぜるため、同じ入力でも再生成結果は変わり得る。

### 8.6 入力値の扱い

- `name`、`theme`、`mood` は `trim()` した値を使用する。
- シード生成にも `trim()` 後の `name`、`theme`、`mood` を使用する。
- `birthDate` は予言本文に必ず含めなくてよい。
- `gender` は未選択なら本文に含めなくてよい。
- 入力値はHTMLとして描画せず、Reactの通常のテキスト描画に任せる。

### 8.7 四行詩の解釈軸生成

`buildInterpretationAxis` は、固定の分類名や分類IDだけを返さない。実際に生成された4週分の詩と、生成時に保持したメタ情報から、AIが四行詩を読み替えるための観点を作る。

参照する情報:

- 4週分の `line`
- 各週の `candidateId`
- 各週の `lineKey`
- 各週の `selectedTerms.symbol`
- 各週の `selectedTerms.place`
- 各週の `selectedTerms.object`
- 各週の `selectedTerms.action`
- 各週の `selectedTerms.toneHint`
- 各週の `selectedTerms.adviceNuance`
- ユーザーの `theme`
- ユーザーの `mood`

出力は、4週の流れと相談テーマへの読み替え方が分かる短い日本語文にする。週ごとの観点と、1ヶ月全体の観点を含める。

例:

```text
第1週は余白と小さな印から未整理の情報を見直す週、第2週は手元の道具と順番から進め方を整える週、第3週は気分の揺れから見方を変える週、第4週は選んだ行動語に沿って小さく締める週として読む。全体として、相談テーマを急いで結論づけず、今月の気分を手がかりに優先順位と次の一手を整える流れとして解釈する。
```

生成ルール:

- 実際に選ばれていない語彙やニュアンスを解釈軸に含めない。
- 週番号の対応を守り、第1週から第4週までの流れとして要約する。
- ユーザーの未来、相手の気持ち、成功失敗を断定する観点にしない。
- 医療、法律、金融など専門判断を代替する観点にしない。
- 固定の分類名はAI用プロンプトに含めない。

## 9. AI用プロンプト生成仕様

### 9.1 生成関数

```ts
export function buildAiPrompt(
  input: ProphecyInput,
  weeks: ProphecyWeek[],
  interpretationAxis: string,
): string;
```

### 9.2 プロンプト構成

AI用プロンプトは以下の順に組み立てる。

1. 依頼文
2. ユーザー入力
3. アプリ内生成の四行詩
4. 四行詩の解釈軸
5. 出力条件
6. 注意事項

### 9.3 プロンプト文面

以下の内容を含める。

```text
あなたは日本語で、占い風テキストを現実的な助言に読み替えるアシスタントです。
以下のユーザー入力と四行詩を読み取り、相談テーマに関係する助言を返してください。

【ユーザー入力】
名前:
生年月日:
性別:
相談テーマ:
今月の気分:

【四行詩】
...

【四行詩の解釈軸】
...

【出力条件】
- 日本語で出力する
- 四行詩の内容を助言の根拠として扱う
- 四行詩の解釈軸を、読み替えの観点として扱う
- ユーザーの相談テーマに直接関係する助言を返す
- 不穏さを煽らず、次に取れる小さな行動へ落とし込む
- 占いや予言として断定せず、解釈の一例として提示する
- 医療、法律、金融など専門判断が必要な領域では、専門家への相談を促す
```

### 9.4 プロンプトの表示

- 改行を保持する。
- ユーザーが全文を選択できる。
- コピーに失敗しても本文が読める。

## 11. デザイン仕様

### 11.1 方向性

古い占い紙面、紙、インク、タイプライター、古い机上、自動筆記を連想する静かなUIにする。

### 11.2 禁止事項

- 公式画像を使わない。
- 漫画コマやアニメ素材を使わない。
- 既存キャラクター画像を使わない。
- 読みにくいほど暗い配色にしない。
- ボタンや入力欄のテキストがはみ出すレイアウトにしない。

### 11.3 レイアウト

- 最大幅は `960px` 前後に制限する。
- スマートフォンでは1カラムにする。
- デスクトップではフォームと結果を適度な余白で並べるか、読み順優先の1カラムにする。
- 入力欄、ボタン、プロンプト本文は幅100%を基本にする。
- 長い相談テーマやプロンプトで横スクロールが出ないようにする。

ページ外枠はMUIの `Container` を使ってよい。細かいレスポンシブ配置はTailwind CSSの `grid`、`gap-*`、`max-w-*`、`px-*`、`py-*` などで指定する。

### 11.4 色と質感

推奨:

- 背景: 暗い木机または黒に近い茶色
- メイン面: 古紙風の淡い紙色
- 文字: 濃いインク色
- アクセント: 深い赤、鈍い金、くすんだ青など少量

CSSだけで紙の質感を表現してよい。画像アセットは必須ではない。

MUIテーマでは、フォームやボタンの基本色を古紙風UIに合うよう調整する。Tailwind CSSでは、背景、紙面風の余白、境界線、影、文字の折り返しを補助する。

## 12. アクセシビリティ仕様

- 入力項目には `label` を関連付ける。
- エラーメッセージは項目の近くに表示する。
- ボタンはキーボードで操作できる。
- 結果生成後もページ全体が読みやすいコントラストを保つ。
- 色だけで必須やエラーを伝えない。

## 13. エラー処理

### 13.1 入力エラー

バリデーションメッセージを表示し、生成しない。

### 13.2 コピーエラー

コピー失敗メッセージを表示し、手動コピー可能な本文を維持する。

### 13.3 生成エラー

テンプレート生成はローカル処理のため原則失敗しない設計にする。万一例外が発生する実装にする場合は、画面に以下を表示する。

```text
生成中に問題が発生しました。入力内容を確認して、もう一度お試しください。
```

## 14. テスト仕様

### 14.1 単体テスト対象

`validation.ts`

- 必須項目が空の場合にエラーを返す。
- 性別が空でもエラーにならない。
- 名前が50文字以内なら通る。
- 名前が51文字以上ならエラーになる。
- 相談テーマと今月の気分が255文字以内なら通る。
- 相談テーマと今月の気分が256文字以上ならエラーになる。
- 今月の気分がプリセット選択肢にない自由入力でも、255文字以内なら通る。

`prophecyGenerator.ts`

- 常に4週分を返す。
- 各週が1行の予言文を持つ。
- `weekNumber` が1から4になる。
- 各週の予言文が空文字ではない。
- 禁止語彙が含まれにくいことを確認する。

`promptBuilder.ts`

- ユーザー入力を含む。
- 4週分の予言を含む。
- 出力条件を含む。
- 専門判断が必要な領域では専門家相談を促す文言を含む。

### 14.2 画面テスト観点

- 必須項目未入力では生成ボタンが押せない。
- バリデーションエラーがある場合は生成ボタンが押せない。
- 必須項目入力後、かつバリデーションエラーがない場合に生成ボタンが押せる。
- 今月の気分のプリセット選択肢を押すと、今月の気分の入力欄が選択文言で埋まる。
- 今月の気分のプリセット選択後も、入力欄を自由に編集できる。
- 今月の気分をプリセット選択肢にない文言で自由入力しても生成できる。
- 必須項目入力後に生成すると結果が表示される。
- 結果には第1週から第4週まで表示される。
- 各週の予言文が1行表示される。
- 入力変更によってバリデーションエラーが発生した場合、再生成ボタンが押せない。
- 有効な入力状態で再生成ボタンを押すと結果生成処理が実行される。
- 同じ入力で再生成した場合でも、予言本文とAI用プロンプト本文が変わる場合がある。
- 入力値を変更して生成した場合、結果が変わる場合がある。
- AI用プロンプトが表示される。
- コピーボタン押下で成功または失敗メッセージが表示される。

### 14.3 レスポンシブ確認

確認幅:

- 375px
- 768px
- 1280px

確認内容:

- 入力欄が画面外にはみ出さない。
- ボタンテキストがはみ出さない。
- 予言文が読みやすい。
- AI用プロンプト欄が横スクロールを発生させない。

## 15. 受け入れ条件

初期実装は以下を満たしたら完了とする。

- 必須入力をすべて入れると予言を生成できる。
- 必須入力が欠けている場合、生成ボタンを押せない。
- バリデーションエラーがある場合、生成ボタンと再生成ボタンを押せない。
- 必須入力がすべて入り、バリデーションエラーがない場合のみ生成ボタンを押せる。
- 今月の気分はプリセット選択肢から入力できる。
- 今月の気分はプリセット選択後も自由に編集できる。
- 今月の気分はプリセット選択肢にない自由入力でも生成に使える。
- 生成結果が常に4週分である。
- 各週が常に1行である。
- 同じ入力でも、再生成で異なる予言本文が生成される場合がある。
- 入力値が変わった場合、異なる予言本文が生成される場合がある。
- AI用プロンプトにユーザー入力、4週分の予言、出力条件が含まれる。
- AI用プロンプトをコピーできる。
- コピー失敗時も手動コピーできる。
- 注意書きが画面内に表示される。
- 公式素材を使っていない。
- スマートフォン幅でレイアウトが崩れない。
