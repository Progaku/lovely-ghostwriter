# ラブリーゴーストライター風 Webアプリ 要件定義

## 1. 概要

HUNTER×HUNTERの「ラブリーゴーストライター」をモチーフにした、個人/限定公開向けの非公式ファンWebアプリを作る。

ユーザーが軽い情報を入力すると、アプリ内テンプレートで「1ヶ月分 / 4週分 / 各週1行」の予言を生成する。さらに、その予言を生成AIへ貼り付けて文章を磨くためのプロンプトも出力する。

本アプリは娯楽用途のファン作品であり、公式素材や公式画像は使用しない。

## 2. 想定ユーザーと公開範囲

- 想定ユーザー: HUNTER×HUNTERを知っている個人ユーザー、友人、限定公開先のユーザー
- 公開範囲: 個人利用または限定公開
- 位置づけ: 非公式ファン作品
- 用途: 占い風の文章生成を楽しむ娯楽アプリ

## 3. コア体験

1. ユーザーが名前、生年月日、相談テーマなどを入力する。
2. 「予言を生成する」操作を行う。
3. アプリが4週分の予言を生成する。
4. 同時に、生成AIへ貼り付けるためのプロンプトを表示する。
5. ユーザーは予言結果を読む、またはAI用プロンプトをコピーして任意の生成AIに貼り付ける。

## 4. 機能要件

### 4.1 入力フォーム

以下の入力項目を持つ。

- 名前: 必須
- 生年月日: 必須
- 性別: 任意
- 相談テーマ: 必須
- 今月の気分: 必須

今月の気分には入力補助として、あらかじめ用意した選択肢を表示する。選択肢を選ぶと今月の気分の入力欄にその文言が入る。選択後も入力欄は編集可能で、自由入力もできる。

必須項目が未入力の場合は、予言を生成できない。

必須項目が未入力、または入力値がバリデーション条件に沿っていない場合は、生成ボタンを押せない状態にする。

### 4.2 予言生成

- 生成結果は1ヶ月分とする。
- 1ヶ月分を4週に分ける。
- 各週は必ず1行の詩的な予言文にする。
- 文章は象徴的、不穏、暗示的な雰囲気にする。
- ただし、現実の危害、病気、死、犯罪、金銭損失などを断定的に予告しない。
- 入力値から生成したシードを使い、生成結果に入力内容が反映されるようにする。
- シードは結果を固定するためではなく、ランダム生成の初期値として扱う。
- 同じ入力でも、再生成時に多少異なる結果になることが望ましい。
- 入力値が変わった場合は、生成結果も変わり得る。

### 4.3 AI用プロンプト生成

予言生成後、以下を含むコピー用プロンプトを表示する。

- ユーザー入力
- アプリ内で生成した4週分の予言
- 生成AIへの依頼内容
- プロンプト制限
  - 日本語で出力する
  - アプリ内で生成した四行詩の内容を読み取り、助言の根拠として扱う
  - ユーザーの悩み入力に直接関係する助言を返す
  - 詩の不穏さを煽らず、ユーザーが次に取れる小さな行動へ落とし込む
  - 占いや予言として断定せず、解釈の一例として提示する
  - 医療、法律、金融など専門判断が必要な領域では、専門家への相談を促す

アプリ自身はAI APIを呼び出さない。ユーザーが任意の生成AIに貼り付ける方式とする。

### 4.4 コピー機能

- AI用プロンプトをクリップボードへコピーできる。
- コピー成功時は短い成功表示を出す。
- コピー失敗時は手動コピーできるよう、プロンプト本文を選択可能な形で表示する。

### 4.5 保存・共有

以下は初期スコープ外とする。

- 結果の永続保存
- 履歴表示
- URL共有
- 画像保存
- SNS投稿連携
- ユーザーアカウント

## 5. 非機能要件

### 5.1 技術構成

- React
- Vite
- TypeScript
- フロントエンド単体SPA

バックエンド、DB、認証、外部AI API連携は初期実装では持たない。

### 5.2 表示言語

- UIは日本語のみ。
- 生成される予言文も日本語のみ。
- AI用プロンプトも日本語のみ。

### 5.3 デザイン方針

ビジュアルは「古い占い紙面」を基本とする。

- 紙
- インク
- タイプライター
- 古い机上
- 自動筆記
- 暗めで静かな雰囲気

公式画像、漫画コマ、アニメ素材、既存キャラクター画像は使用しない。

### 5.4 注意書き

画面内に最小限の注意書きを表示する。

含める内容:

- 非公式ファン作品であること
- 娯楽用途であること
- 重要な判断には使用しないこと

## 6. データ構造案

### 6.1 入力

```ts
type ProphecyInput = {
  name: string;
  birthDate: string;
  gender?: string;
  theme: string;
  mood: string;
};
```

### 6.2 出力

```ts
type ProphecyWeek = {
  weekNumber: 1 | 2 | 3 | 4;
  line: string;
};

type ProphecyResult = {
  weeks: ProphecyWeek[];
  interpretationAxis: string;
  aiPrompt: string;
};
```

4週分の予言生成では、固定の4週セットだけを選ばない。第1週の候補を選び、その候補が持つ次行候補から第2週を選び、第2週の候補が持つ次行候補から第3週を選ぶ形で、自然につながる4週分を `weeks` として組み立てる。

`interpretationAxis` は、固定の分類名ではなく、実際に選ばれた4行、各行のテンプレート候補、差し込まれた象徴語・道具語・行動語、週ごとの助言ニュアンスをもとに組み立てる。

## 7. 画面要件

### 7.1 メイン画面

- アプリ名または能力名を想起できるタイトル
- 入力フォーム
- 生成ボタン
- 注意書き

### 7.2 結果表示

- 4週分の予言を週ごとに表示する。
- 各週の予言文は読みやすく表示する。
- AI用プロンプトを表示する。
- AI用プロンプトのコピーボタンを表示する。
- 再生成ボタンを表示する。

## 8. バリデーション

- 名前が空の場合はエラー。
- 生年月日が空の場合はエラー。
- 相談テーマが空の場合はエラー。
- 今月の気分が空の場合はエラー。
- 入力値の長さはUIが崩れない範囲に制限する。
- 必須項目が未入力、または入力上限を超えている場合は、生成ボタンを押せない。

入力上限:

- 名前: 50文字
- 性別: 男性/女性/その他 から選ぶ
- 相談テーマ: 255文字
- 今月の気分: 255文字
  - 選択肢を用意し、選択肢の入力と自由入力の両方ができるようにする
  - 選択肢を選ぶと入力欄の値をその選択肢で埋める

## 9. テスト観点

### 9.1 入力フォーム

- 必須項目がすべて入力されている場合、生成できる。
- 必須項目が空の場合、生成ボタンを押せない。
- 入力値がバリデーション条件に沿っていない場合、生成ボタンを押せない。
- 性別が未入力でも生成できる。
- 今月の気分の選択肢を選ぶと、今月の気分の入力欄が選択内容で埋まる。
- 今月の気分は選択肢を使わず自由入力しても生成できる。
- 長い入力でもUIが崩れない。

### 9.2 予言生成

- 常に4週分の結果が生成される。
- 各週が必ず1行になる。
- 同じ入力でも、再生成で異なる表現が出る場合がある。
- 入力値が変わると、異なる表現が出る場合がある。
- 危険な断定表現が出にくい。

### 9.3 AI用プロンプト

- ユーザー入力が含まれる。
- 生成済みの4週分予言が含まれる。
- プロンプト制限が含まれる。
- プロンプト制限に、四行詩とユーザーの悩み入力を元に助言するためのガイドラインが含まれる。
- コピー操作ができる。

### 9.4 レスポンシブ表示

- スマートフォン幅で入力フォームが崩れない。
- スマートフォン幅で予言文が読みやすい。
- ボタン内テキストがはみ出さない。
- デスクトップ幅で紙面風の余白と読みやすさが保たれる。

## 10. 初期スコープ外

- AI API呼び出し
- APIキー管理
- サーバーサイド処理
- データベース保存
- 認証
- 多言語対応
- 画像保存
- URL共有
- SNS投稿
- 公式素材の利用

## 11. 実装時の前提

- リポジトリは空の状態から開始する。
- 初期実装はフロントエンド単体で完結させる。
- 生成AIとの連携は「プロンプトを作る」までに限定する。
- 作品名や能力名への言及はファン作品の文脈に留める。
- 公開時は非公式ファン作品であることを明示する。
